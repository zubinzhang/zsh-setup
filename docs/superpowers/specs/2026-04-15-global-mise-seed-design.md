# Global Mise Seed Design

## Goal

Make a fresh `install.sh` run seed a usable global `mise` tool configuration from the repository's root `mise.toml` so commands like `node`, `go`, and `python` work from `~` immediately after install, while preserving any later user edits to `~/.config/mise/config.toml`.

## Problem Summary

The repository currently has two separate `mise` configuration scopes:

- the repo root [`mise.toml`](/Users/zubin/workspace/github.com/zubinzhang/zsh-setup/mise.toml), which defines the full toolset used inside this repository
- the managed global `~/.config/mise/config.toml`, which is generated by [`home/run_onchange_before_setup-mise-config.sh`](/Users/zubin/workspace/github.com/zubinzhang/zsh-setup/home/run_onchange_before_setup-mise-config.sh) and only includes `helm`, `kubectl`, and `starship`

That means bootstrap installs tool versions for this repo, but a user standing in `~` still hits `mise` shims without global versions for `node`, `go`, `python`, `bun`, or `yarn`.

## Design

### Single Seed Source

Keep the repo root [`mise.toml`](/Users/zubin/workspace/github.com/zubinzhang/zsh-setup/mise.toml) as the single source of truth for the initial global tool list. The install flow should use that file directly to create the first global `mise` config.

The generated global file should preserve the declared versions exactly, including fixed versions such as `node = "24.14.0"` and symbolic values such as `helm = "latest"`.

### First-Install-Only Seeding

Bootstrap should seed `~/.config/mise/config.toml` only when that file does not already exist.

Behavior:

- no existing global config: copy the repo `mise.toml` to `~/.config/mise/config.toml`
- existing global config: leave it untouched

This keeps install behavior predictable for fresh machines while preserving later user customization.

### Ownership Boundary

The seeded global file is an installation default, not an always-managed file.

After the first install:

- `chezmoi apply`
- `sync`
- future upgrades

must not overwrite the user's global `mise` config.

To enforce that boundary, the repository should stop generating `~/.config/mise/config.toml` through a `run_onchange` chezmoi hook. The install/bootstrap path becomes the only place that writes the initial file.

### Installation Flow

The install flow should:

1. ensure `mise` itself is installed
2. copy repo `mise.toml` to global `~/.config/mise/config.toml` if missing
3. run `mise install` in the repo so repo-local tools are installed

Seeding the global config before `mise install` keeps the bootstrap flow internally consistent and ensures the user's shell can resolve the same tool set outside the repo after install completes.

## Non-Goals

- continuously syncing repo `mise.toml` changes into the user's global config
- merging repo defaults into an existing user-managed `~/.config/mise/config.toml`
- using `mise sync`, which is for importing versions from other version managers and does not solve this workflow for `go`
- replacing the repo-local `mise.toml` as the source for repository tasks

## Testing

Add regression coverage for:

- fresh bootstrap seeds global `mise` config from repo `mise.toml`
- seeded config includes fixed-version tools like `go`, `node`, `python`, `bun`, and `yarn`
- seeded config still preserves symbolic values like `latest`
- bootstrap does not overwrite an existing user `~/.config/mise/config.toml`
- repository source no longer contains a chezmoi `run_onchange` writer for global `mise` config

## User-Visible Outcome

After a fresh install, opening a shell in `~` should allow `node`, `go`, `python`, `bun`, `helm`, `kubectl`, `starship`, and `yarn` to resolve through `mise` without requiring the user to run `mise use -g ...` manually.
